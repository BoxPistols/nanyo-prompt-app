## はじめに

山形県南陽市（人口約3万人）が、800件近い生成AIプロンプトをCC BY 4.0ライセンスでGitHubに公開しています。

https://github.com/nanyo-line/prompt

南陽市DX推進主幹・佐野毅氏（[@ichigonme](https://x.com/ichigonme)）が主導する「一発OK!! 市民も使える！生成AI活用実例集」というプロジェクトで、文章作成・業務改善・アイデア創出・プログラミングなど9カテゴリのプロンプトが、誰でも自由に使える形で公開されています。

ただ、実際に使おうとすると困ることがありました。

- プロンプトが個別のHTMLページに散在していて、目的のものを探しにくい
- コピーして使うまでの手順が多い
- 全体にどんなプロンプトがあるか俯瞰できない

せっかくの宝の山なのに、もったいない。この課題を解決するために、Webアプリ「南陽市DX Prompts」を開発しました。

https://nanyo-prompt.vercel.app/

南陽市の確認のもと、個人が開発・運営しているアプリです。本記事ではアプリの機能紹介から、日本語検索エンジンの設計、データ自動同期パイプラインまで、開発の記録をまとめます。

## アプリでできること

### 4つの検索モードで「見つからない」を解消

このアプリの核心は検索です。数百件のプロンプトから目的のものを見つけるために、4つの検索モードを用意しました。

| モード | 特徴 | 使いどころ |
|:---|:---|:---|
| キーワード | タイトル・カテゴリへの厳密ANDマッチ | 探したいものが明確なとき |
| 意図検索 | 同義語・関連語を自動展開して全文検索 | 「こんな感じの」で探したいとき |
| あいまい | タイプミスや表記ゆれに対応（N-gram類似度） | うろ覚えのキーワードで探すとき |
| スマート（デフォルト） | 上記3つを統合 | 迷ったらこれ |

検索窓に「メール」と入れれば、メール文面作成系のプロンプトがすぐ出てきます。「mail」でも「めーる」でも同じ結果になるのは、後述する日本語正規化のおかげです。

### ワンクリックでAIツールへ送信

見つけたプロンプトは、ChatGPT・Gemini・Claudeにワンクリックで送れます。

1. プロンプトを選ぶ
2. 変数がある場合はフォームに入力（「宛先」「用件」など）
3. 「コピーしてAIで使う」をクリック

AIツールが新しいタブで開き、プロンプトがクリップボードにコピーされた状態になります。生成AIを使ったことがない人でも30秒で最初の体験ができる設計にしました。

### カスタムプロンプトの作成

南陽市のプロンプトに加えて、自分だけのプロンプトも作成・管理できます。

- シンプル入力: テキストエリアに自由記述
- セクション入力: 目的・役割・前提条件・実行指示・ルール・出力形式・補足・変数設定の8フィールドで構造的に記述
- テンプレート: 文章要約、コードレビュー、メール作成などから選んでカスタマイズ

`{変数名}` の構文を使うと、実行時に入力フォームが自動生成されます。

### その他

- カテゴリフィルタ - 9つの大カテゴリで絞り込み
- お気に入り - よく使うプロンプトをブックマーク
- エクスポート/インポート - カスタムプロンプトとお気に入りをJSONで書き出し・読み込み（追加/マージ/上書きの3モード）
- ダークモード - ショートカット`D`キーで切り替え
- キーボードショートカット - `Cmd/Ctrl+K`で検索フォーカス、矢印キーでカード移動
- オンボーディングツアー - 初回利用時にステップバイステップで案内

## プライバシーファースト：サーバーなし設計

このアプリにはバックエンドサーバーがありません。お気に入り・カスタムプロンプト・テーマ設定・オンボーディング進捗──すべてブラウザの`localStorage`に保存されていて、外部に送信されることはありません。

データ移行が必要な場合は、エクスポート機能でJSONファイルをダウンロードし、別のブラウザでインポートできます。

## 技術的な見どころ：日本語検索を"ちゃんと"動かす

ここからはエンジニア向けに、技術的な設計判断を掘り下げます。

### テキスト正規化 ── 表記ゆれとの戦い

日本語検索で最初にぶつかる壁が表記ゆれです。ユーザーが「メール」と入力しても、データ側には「めーる」「MAIL」「ＡＩ」「ａｉ」といった表記が混在しています。

アプリの`normalize()`関数は3段階の正規化を行います。

```js
/** カタカナ → ひらがな */
const katakanaToHiragana = (str) =>
  str.replace(/[\u30A1-\u30F6]/g, (ch) =>
    String.fromCharCode(ch.charCodeAt(0) - 0x60)
  );

/** 全角英数 → 半角、カタカナ → ひらがな、小文字化 */
const normalize = (str) => {
  if (!str) return "";
  let s = String(str);
  // 全角英数 → 半角（Unicodeオフセット 0xFEE0）
  s = s.replace(/[Ａ-Ｚａ-ｚ０-９]/g, (ch) =>
    String.fromCharCode(ch.charCodeAt(0) - 0xfee0)
  );
  s = katakanaToHiragana(s);
  return s.toLowerCase().trim();
};
```

十数行のコードですが、これがあるかないかで日本語検索の精度は劇的に変わります。「メール」「めーる」「MAIL」「ｍａｉｌ」がすべて同じ検索語として扱われるようになります。

### スコアリング ── フィールド別の重み付け

検索結果の順序を決めるスコアリングには、フィールドごとの重み付けを採用しています。

```js
const FIELD_WEIGHTS = {
  title: 10,   // タイトル一致を最優先
  c1: 5,       // 大カテゴリ
  c2: 4,       // 中カテゴリ
  c3: 3,       // 小カテゴリ
  sub: 2,      // サブカテゴリ
  tag: 2,       // タグ
  id: 1,       // ID
  content: 1,  // プロンプト本文（意図/スマートモードのみ）
};
```

さらに完全一致は重み×3倍、部分一致は×2倍で加算し、全トークンが含まれる場合は1.5倍のANDボーナスを付けます。「だいたい合ってる」ではなく「まさにこれ」が上に来るようにするためです。

### 意図検索 ── 同義語の双方向展開

意図検索（Intent Search）は、手作業で構築した同義語マップ（`INTENT_MAP`）で検索語を自動展開します。

```js
const INTENT_MAP = {
  "メール": ["ビジネスメール", "メッセージ", "コミュニケーション",
             "メルマガ", "連絡", "返信", "挨拶"],
  "書く":   ["文章作成", "作成", "文書", "ライティング", "執筆"],
  "要約":   ["まとめ", "サマリー", "概要", "ダイジェスト"],
  // ... 約60キーワード、14の意味領域にわたって定義
};
```

ポイントは逆引きマップの自動生成です。

```js
// 同義語 → 元キー群の逆引きマップを構築
const REVERSE_INTENT_MAP = (() => {
  const reverse = {};
  Object.entries(INTENT_MAP).forEach(([key, synonyms]) => {
    synonyms.forEach((syn) => {
      const normSyn = normalize(syn);
      if (!reverse[normSyn]) reverse[normSyn] = new Set();
      reverse[normSyn].add(normalize(key));
    });
  });
  return reverse;
})();
```

「メール」→「ビジネスメール」だけでなく、「ビジネスメール」→「メール」の双方向で関連語検索が動きます。辞書を手動で維持するとき、片方だけ書けばいいので保守しやすくなります。

### あいまい検索 ── バイグラムのDice係数

あいまい検索ではバイグラム（2文字組）のDice係数で類似度を計算します。

```js
const bigramSimilarity = (a, b) => {
  const setA = bigrams(a);
  const setB = bigrams(b);
  if (setA.size === 0 && setB.size === 0) return 1;
  if (setA.size === 0 || setB.size === 0) return 0;
  let intersection = 0;
  setA.forEach((g) => { if (setB.has(g)) intersection++; });
  return (2 * intersection) / (setA.size + setB.size);
};
```

たとえば「プレゼン」と「プレゼンテーション」の場合:

- 「プレゼン」のバイグラム: `{プレ, レゼ, ゼン}` → 3個
- 「プレゼンテーション」のバイグラム: `{プレ, レゼ, ゼン, ンテ, テー, ーシ, ショ, ョン}` → 8個
- 共通: 3個
- Dice係数: `2 × 3 / (3 + 8) ≈ 0.55`

さらに長いテキスト中の部分一致を拾うため、スライディングウィンドウで最大類似度を求める`partialBigramSimilarity`も用意しています。タイトルの一部にだけ該当語が含まれるケースに対応するためです。

### スマート検索 ── 3モードの統合

デフォルトのスマート検索は、3つのモードのスコアを異なる重みで足し合わせます。

| モード | 重み |
|:---|:---|
| キーワード | ×1.0（基本） |
| 意図検索 | ×0.8（補助） |
| あいまい | ×0.6（補完） |

正確なキーワード一致を最優先しつつ、関連語やタイプミスにも対応できるバランスです。外部ライブラリを使わず、この検索ロジック全体が`src/utils/search.js`の1ファイルに収まっています。

## データの自動同期 ── GitHub Actions × Python

南陽市のプロンプトデータは日々更新される可能性があります。手動同期は現実的でないため、GitHub Actionsによる自動同期パイプラインを構築しました。

### 同期の仕組み

```
毎朝 9:00 (JST) に GitHub Actions が起動
  ↓
GitHub API で nanyo-line/prompt のファイルツリーを取得
  ↓
SHA ハッシュを前回（upstream_meta.json）と比較
  → 変更・追加・削除されたファイルのみ検出
  ↓
変更があった HTML ページだけを取得・パース
  ↓
raw_data.json / contents.json を更新して自動コミット
```

ポイントはSHAベースの差分検出です。全件を毎回取得するのではなく、変更があったファイルだけを処理するので、効率的かつGitHub APIのレート制限にも優しい設計です。

### HTML本文の抽出

南陽市のHTMLページはフォーマットが統一されていないため、複数の候補要素を優先順位で試行します。

```python
candidates = [
    soup.find("div", id="text_copy"),
    soup.find("textarea", id="copy_text"),
    soup.find("div", class_="prompt-text"),
    soup.find("div", class_="box_txt"),
    soup.find("pre"),
    soup.find("code"),
]
```

差分同期スクリプト（`sync_upstream.py`）はBeautifulSoup4で実装し、初回一括取得用のスクリプト（`main.py`）は10件ごとに中間保存する再開対応付きです。

### クライアント側のデータマージ

アプリ側でも、データ更新時にユーザーのカスタムプロンプトを壊さないためのマージ戦略を`syncLogic.js`に実装しています。

- ソースプロンプト（`isUser: false`）→ 最新データで上書き
- ユーザー作成プロンプト（`isUser: true`）→ そのまま保持
- 削除されたプロンプト → お気に入りからも自動クリーンアップ

このマージロジックにはVitestで10ケースのユニットテスト（エッジケース含む）を書いています。「ソースが大幅に減った場合」「IDが完全に入れ替わった場合」なども網羅しています。

## データ構造の工夫

プロンプトデータは2つのJSONで管理しています。

`raw_data.json` ── メタデータ（75KB）。配列の配列で圧縮されています:

```json
[284, "季節と状況に応じたビジネス定型文作成", "284", 7, 25, 44, 34, 4, 0, "2025-01-29"]
// [ID, タイトル, LinkID, C1, C2, C3, Sub, Tag, IsNew, CreatedAt]
```

カテゴリはインデックス参照（`RAW.c1[d[3]]`で展開）にすることで、重複する文字列を排除してサイズを抑えています。

`contents.json` ── プロンプト本文（5.3MB）。IDをキーとした辞書です:

```json
{
  "284": "#284 季節と状況に応じたビジネス定型文作成\n\n目的・ねらい\r\n..."
}
```

5.3MBと大きいですが、クライアントサイド全文検索のためにバンドルに含めています。Viteのビルドで圧縮されるため、実際の転送量はそこまで大きくなりません。

あえてUIライブラリを使わない選択をしました。9カテゴリ×ライト/ダークの18パターンのカラーシステム、グラスモーフィズムのヘッダー（`backdrop-filter: blur(16px)`）など、アプリ固有のデザインを柔軟に実装するためです。

アーキテクチャも割り切っています。ルーターなし、状態管理ライブラリなし。UIは`App.jsx`の1ファイル（約1,700行）に集約し、検索ロジックだけ`search.js`に分離しています。小規模アプリにおけるこの「巨大単一コンポーネント」方式は賛否あると思いますが、全体を把握しやすく、個人開発では回しやすいスタイルです。

### 細かいこだわり

いくつか地味だけど効いている実装を紹介します。

IME対応のデバウンス ── 日本語入力中に検索が暴発しないよう、`compositionstart`/`compositionend`イベントで制御しています:

```jsx
onCompositionStart={() => { isComposingRef.current = true; }}
onCompositionEnd={(e) => {
  isComposingRef.current = false;
  setQuery(e.target.value);
  setDebouncedQuery(e.target.value);
}}
```

モバイルキーボード対応 ── `visualViewport` APIで`--keyboard-offset` CSS変数を動的に更新し、ソフトキーボードが表示されたときのレイアウト崩れを防止しています。

キーボードナビゲーション ── グリッド表示でのカード間移動で、列数を`getComputedStyle`から動的に取得しています。画面幅が変わっても正しく上下移動できます。

## 「市民も使える」をどう実装するか

南陽市のプロンプト集のタイトルは「市民も使える！」です。

技術的に動くだけでは足りなくて、生成AIに触れたことがない人にも使えるUIが必要でした。具体的に何をしたか。

- オンボーディングツアー: 初回アクセス時にステップバイステップで案内。プロンプト実行モーダル内にも個別ガイドを用意
- 変数の自動フォーム化: プロンプト内の`{宛先}`や`{用件}`が自動で入力欄になる。テキストを手で書き換える必要がない
- 視覚的なカテゴリ分類: 色分けで、文字を読まなくてもカテゴリが判別できる
- レスポンシブ対応: スマートフォンでもストレスなく操作可能

検索窓に「メール」と打ち、プロンプトを選び、宛先と用件を入力し、ボタンを押す。この30秒の体験を、いかにスムーズにするかが設計の軸でした。
